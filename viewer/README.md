# Melkor · SparkJS Splat Viewer

A self-contained web viewer for 3D Gaussian Splats built on
[SparkJS](https://sparkjs.dev) (World Labs) + THREE.js, with a curated set of
**city / area** scenes, camera controls, and a Playwright render-test suite.

It is a quick way to inspect the `.ply` / `.spz` files that
[Melkor](../README.md) produces or converts — for example, the output of
[scene completion](../docs/SCENE_COMPLETION.md) — since SparkJS reads every
splat format Melkor emits.

## Scenes

All scenes are real-world *city / area* captures, spanning every container format
SparkJS auto-detects:

| Scene                | Format       | Splats | Source | Description |
|----------------------|--------------|-------:|--------|-------------|
| Snowy Street         | SPZ          |  ~982K | sparkjs.dev | snowy urban street |
| Valley               | SPZ          |   500K | sparkjs.dev | canyon / valley landscape |
| Sutro Tower (SF)     | SOG (`.zip`) |  ~2.0M | sparkjs.dev | San Francisco cityscape (SH degree 3) |
| Rail Yard            | SPLAT        |  ~1.0M | huggingface `cakewalk/splat-data` | outdoor rail-yard area |
| Distant Igloo        | SPZ          |   347K | sparkjs.dev | outdoor igloo area |
| Igloo · PLY (melkor) | PLY          |   347K | generated by `melkor` from the SPZ | same igloo as standard 3DGS `.ply` |

Together these cover **all four** splat containers SparkJS auto-detects (SPZ,
SOG, SPLAT, PLY). The PLY scene is optional — it is emitted by Melkor's own
`SPZ → PLY` converter to both exercise Spark's `.ply` path and sanity-check
Melkor's output end to end. At boot the viewer probes each file with a `HEAD`
request and hides scenes whose files are absent.

The files themselves are **not committed** (the five downloads total ~82 MB;
the generated PLY adds ~24 MB); fetch them with the script below.

## Setup

```bash
cd viewer
./fetch-assets.sh                  # downloads vendor libs (three, spark) + splat scenes
npm install                        # only needed to run the Playwright tests
npx playwright install chromium    # only needed to run the Playwright tests
```

`fetch-assets.sh` is idempotent and integrity-checked: every artifact is
verified against a pinned SHA-256 on each run, and a mismatch aborts the
script. It populates two gitignored directories:

- `vendor/` — `three.module.js`, `three.core.js`, `addons/postprocessing/Pass.js`, `spark.module.js`
- `public/splats/` — the five downloaded scenes (and the generated `.ply`, if Melkor is built)

To enable the **PLY** scene, build Melkor once, then re-run the fetch script — it
converts `distant-igloo.spz → distant-igloo.ply` automatically:

```bash
(cd .. && mkdir -p build && cd build && cmake .. && make melkor -j)
./fetch-assets.sh      # now also emits public/splats/distant-igloo.ply
```

## Run

```bash
npm run serve                                  # Bun static server (serve.js)
python3 -m http.server 8771 --bind 127.0.0.1   # alternative: any static server works
```

Open <http://127.0.0.1:8771/> (or `/index.html`). `serve.js` reads `PORT`,
`HOST`, and `QUIET=1` from the environment, serves correct MIME types with
HTTP range support, and rejects path traversal outside `viewer/` with a
separator-aware containment check that is correct on every platform,
including Windows.

### Controls

| Input | Action |
|-------|--------|
| Scene buttons (top-right) | switch scene |
| `Front` / `Side` / `Top` / `3/4` | jump to a named camera feed (disables auto-orbit) |
| `Orbit` | toggle auto-rotate |
| `Reframe` | recompute framing for the current scene and resume orbiting |
| Drag | rotate — enters manual mode (SparkJS `SparkControls` / `PointerControls`) |
| Scroll | dolly in/out — enters manual mode |
| `WASD` / arrows | fly (manual mode; `FpsMovement`) |
| `Q` / `E` | move down / up (manual mode) |
| `Shift` / `CapsLock` / `Ctrl` | movement speed ×5 / ×10 / ×0.2 |

Keyboard flight is active only in manual mode — drag or scroll on the canvas
first. The HUD (top-left) shows the live scene, format, splat count, FPS,
camera position, and control mode (orbit / manual / static).

### Desktop app (Tauri)

The same viewer ships as a native desktop app via a Tauri v2 shell (`src-tauri/`,
WKWebView on macOS / WebKitGTK on Linux / WebView2 on Windows). Dev mode launches
the Bun static server (`serve.js`) and points the window at it:

```bash
bun run app          # = cargo tauri dev   — opens the desktop window
bun run app:build    # = cargo tauri build — bundles a .app / .dmg / etc.
```

Requires the Rust toolchain + Tauri CLI (`cargo install tauri-cli`). `app:build`
stages a clean `dist/` (just `index.html` + `vendor/` + `public/`) via
`stage-dist.js`, so the bundle never embeds `node_modules/` or `src-tauri/target/`.
Run `./fetch-assets.sh` first so the scenes are present.

## How it works

- **Imports** are resolved by a browser `importmap` to the local `vendor/` files, so
  the viewer runs fully offline once assets are fetched. SparkJS bundles its sort
  worker + LoD WASM inline (no extra network fetches).
- A single `SparkRenderer` is added to the THREE scene; each scene is one
  `SplatMesh({ url })`. A 180° X-flip (`quaternion.set(1,0,0,0)`) maps the
  Y-down/Z-forward training convention into THREE's Y-up space.
- **Auto-framing** (`computeBounds`): Spark's `getBoundingBox()` returns `NaN` for
  packed/compressed scenes, so bounds are derived from splat centers — keeping only
  the more-opaque half of a sample (to ignore atmospheric "haze" and extreme
  floaters), then using a median center + 90th-percentile-distance radius. This
  frames object-centric and immersive 360 captures alike.
- **Progressive streaming**: on the first load (nothing on screen yet) the `SplatMesh`
  is added to the scene *before* the download completes, so Spark renders splats as
  they stream in instead of a blank screen until 100%; the overlay drops its dim/blur
  to a floating progress card so the build-up is visible. A scene *switch* keeps the
  current scene until the new one is ready, so two scenes never overlap mid-stream.
  Spark 2.1 also supports `.RAD` progressive assets, continuous LOD, and virtual splat
  paging for very large worlds — see [../docs/STREAMING.md](../docs/STREAMING.md).
- A monotonic **load token** guards `loadScene` so a superseded load (e.g. the boot
  auto-load racing a user click) never clobbers viewer state.
- `window.__viewer` exposes a small automation API (`load`, `setView`, `setAngles`,
  `setOrbit`, `getStats`, `waitRendered`) used by the tests. `waitRendered()`
  resolves only after the newly loaded scene has been swapped in **and** rendered
  several frames (the frame counter resets on swap), so a screenshot taken after
  it resolves shows the new scene, not the previous one.

## Tests

```bash
npm test               # playwright test (reuses a running server, else starts one via Bun)
npm run test:headed    # same suite with a visible browser window
```

The suite (`tests/viewer.spec.js`) loads every scene in headless Chromium and asserts:

- each splat asset is served (HTTP 200 on a `HEAD` request),
- WebGL2 is available and the scene loads with the expected format + a plausible splat count,
- each of the four camera feeds renders a **non-blank** frame (framebuffer readback:
  non-background pixel fraction + luminance variance) and writes a screenshot to
  `screenshots/`,
- camera controls actually move the viewpoint (named views produce distinct camera
  positions, auto-orbit advances over time, a 180° azimuth sweep via `setAngles`
  displaces the camera).

Screenshots land in `screenshots/` (gitignored) — one per scene × view, up to
6 scenes × 4 views; the optional PLY scene is skipped when its file is absent.

## Adding a scene

1. Add a `path|url|sha256` entry to `fetch-assets.sh` (`public/splats/<file>`
   plus the pinned SHA-256 of the download).
2. Add an entry to the `SCENES` array in `index.html` (`id`, `label`, `file`,
   `fmt`, `note`). Missing files are hidden automatically by the boot-time
   `HEAD` probe.
3. (Optional) add it to `SCENES` in `tests/viewer.spec.js` (`id`, `fmt`, `min`
   splat count, `file`) to cover it in CI.
