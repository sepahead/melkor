# Melkor · SparkJS Splat Viewer

A self-contained web viewer for 3D Gaussian Splats built on
[SparkJS](https://sparkjs.dev) (World Labs) + THREE.js, with a curated set of
**city / area** scenes, camera controls, and a Playwright render-test suite.

It's a handy way to eyeball `.ply` / `.spz` / `.sog` files that Melkor produces or
converts — SparkJS reads every format Melkor emits.

![scenes](screenshots/sutro-front.png)

## Scenes

All scenes are real-world *city / area* captures, spanning every container format
SparkJS auto-detects:

| Scene            | Format      | Splats | Source | What it is |
|------------------|-------------|-------:|--------|------------|
| Snowy Street     | SPZ         |   982K | sparkjs.dev | snowy urban street |
| Valley           | SPZ         |   500K | sparkjs.dev | canyon / valley landscape |
| Sutro Tower (SF) | SOG (`.zip`)|  2.0M  | sparkjs.dev | San Francisco cityscape (SH degree 3) |
| Rail Yard        | SPLAT       |  1.0M  | huggingface `cakewalk/splat-data` | outdoor rail-yard area |
| Distant Igloo    | SPZ         |   347K | sparkjs.dev | outdoor igloo area |
| Igloo · PLY      | PLY         |   347K | generated by `melkor` from the SPZ | same igloo as standard 3DGS `.ply` |

That's **all four** splat containers SparkJS auto-detects (SPZ, SOG, SPLAT, PLY).
The PLY is optional — it's emitted by Melkor's own `SPZ → PLY` converter to both
exercise Spark's `.ply` path and sanity-check Melkor's output end to end.

The files themselves are **not committed** (they total ~82 MB); fetch them with the
script below.

## Setup

```bash
cd viewer
./fetch-assets.sh      # downloads vendor libs (three, spark) + splat scenes (idempotent)
npm install            # only needed to run the Playwright tests
```

`fetch-assets.sh` populates two gitignored directories:

- `vendor/` — `three.module.js`, `three.core.js`, `addons/postprocessing/Pass.js`, `spark.module.js`
- `public/splats/` — the five downloaded scenes (and the generated `.ply`, if Melkor is built)

To enable the **PLY** scene, build Melkor once, then re-run the fetch script — it
converts `distant-igloo.spz → distant-igloo.ply` automatically:

```bash
(cd .. && mkdir -p build && cd build && cmake .. && make melkor -j)
./fetch-assets.sh      # now also emits public/splats/distant-igloo.ply
```

## Run

```bash
python3 -m http.server 8771 --bind 127.0.0.1   # or: npm run serve
```

Open <http://127.0.0.1:8771/index.html>.

### Controls

| Input | Action |
|-------|--------|
| Scene buttons (top-right) | switch scene |
| `Front` / `Side` / `Top` / `3/4` | jump to a named camera feed |
| `Orbit` | toggle auto-rotate |
| `Reframe` | recompute framing for the current scene |
| Drag | rotate (SparkJS `PointerControls`) |
| Scroll | dolly in/out |
| `WASD` / arrows, `Q`/`E` | fly / roll (SparkJS `FpsMovement`) |

The HUD (top-left) shows the live scene, format, splat count, FPS and camera position.

### Desktop app (Tauri)

The same viewer ships as a native desktop app via a Tauri v2 shell (`src-tauri/`,
WKWebView on macOS / WebKitGTK on Linux / WebView2 on Windows). Dev mode launches
the bun static server (`serve.js`) and points the window at it:

```bash
bun run app          # = cargo tauri dev   — opens the desktop window
bun run app:build    # = cargo tauri build — bundles a .app / .dmg / etc.
```

Requires the Rust toolchain + Tauri CLI (`cargo install tauri-cli`). `app:build`
stages a clean `dist/` (just `index.html` + `vendor/` + `public/`) via
`stage-dist.js`, so the bundle never embeds `node_modules/` or `src-tauri/target/`.
Run `./fetch-assets.sh` first so the scenes are present.

## How it works

- **Imports** are resolved by a browser `importmap` to the local `vendor/` files, so
  the viewer runs fully offline once assets are fetched. SparkJS bundles its sort
  worker + LoD WASM inline (no extra network fetches).
- A single `SparkRenderer` is added to the THREE scene; each scene is one
  `SplatMesh({ url })`. A 180° X-flip (`quaternion.set(1,0,0,0)`) maps the
  Y-down/Z-forward training convention into THREE's Y-up space.
- **Auto-framing** (`computeBounds`): Spark's `getBoundingBox()` returns `NaN` for
  packed/compressed scenes, so bounds are derived from splat centers — keeping only
  the more-opaque half of a sample (to ignore atmospheric "haze" and extreme
  floaters), then using a median center + 90th-percentile-distance radius. This
  frames object-centric and immersive 360 captures alike.
- A monotonic **load token** guards `loadScene` so a superseded load (e.g. the boot
  auto-load racing a user click) never clobbers viewer state.
- `window.__viewer` exposes a small automation API (`load`, `setView`, `setAngles`,
  `setOrbit`, `getStats`, `waitRendered`) used by the tests.

## Tests

```bash
npm test               # playwright test (reuses a running server, else starts one)
```

The suite (`tests/viewer.spec.js`) loads every scene in headless Chromium and asserts:

- the splat asset is served and non-empty,
- WebGL2 is available and the scene loads with the expected format + a plausible splat count,
- each of the four camera feeds renders a **non-blank** frame (framebuffer readback:
  non-background pixel fraction + luminance variance) and writes a screenshot to
  `screenshots/`,
- camera controls actually move the viewpoint (named views differ, auto-orbit
  advances over time, opposite azimuths straddle the target).

Screenshots land in `screenshots/` (gitignored) — 5 scenes × 4 views.

## Adding a scene

1. Add the download to `fetch-assets.sh` (`public/splats/<file>`).
2. Add an entry to the `SCENES` array in `index.html` (`id`, `label`, `file`, `fmt`, `note`).
3. (Optional) add it to `SCENES` in `tests/viewer.spec.js` to cover it in CI.
