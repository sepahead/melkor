# Migration: standalone GLOMAP → COLMAP `global_mapper`

## What changed

Global structure-from-motion has moved. It used to require the separate GLOMAP program
(`github.com/colmap/glomap`); it is now a first-class command inside COLMAP itself:

```text
colmap global_mapper
```

This was verified against COLMAP's `main` branch, where the CLI registers the command directly
(`commands.emplace_back("global_mapper", &colmap::RunGlobalMapper)`). Installing a separate
GLOMAP checkout is therefore obsolete, and Melkor no longer does it (release blocker P0-14).

## Why Melkor stopped installing standalone GLOMAP

The previous `scripts/setup_glomap.sh` did a mutable clone of the standalone repository:

```bash
git clone https://github.com/colmap/glomap.git
git pull origin main   # tracking a moving branch
```

That is exactly the pattern the v2 hardening program forbids: a build that tracks an upstream
branch at run time is not reproducible, and the tool it installed is now the deprecated path.
The supported route is a **pinned** COLMAP build whose `global_mapper` command has been verified
to exist, driven through the reproducible adapter runner rather than an ad hoc shell script.

## What to run instead

You need a COLMAP build recent enough to include `global_mapper`. Confirm it:

```bash
colmap global_mapper --help
```

If that command is missing, your COLMAP predates the integration; update it. Do **not** fall back
to the standalone GLOMAP repository.

### Before (deprecated)

```bash
# Standalone GLOMAP, cloned and built by setup_glomap.sh
./glomap mapper \
  --database_path database.db \
  --image_path images/ \
  --output_path sparse/
```

### After (COLMAP global mapper)

```bash
# Feature extraction and matching, then global mapping, all through COLMAP.
colmap feature_extractor --database_path database.db --image_path images/
colmap exhaustive_matcher --database_path database.db
colmap global_mapper \
  --database_path database.db \
  --image_path images/ \
  --output_path sparse/
```

The output is a standard COLMAP sparse model in `sparse/`, the same format the incremental
mapper produces, so anything downstream that consumed a COLMAP model continues to work
unchanged.

## An honest note on speed

Global mapping and incremental mapping make different robustness and precondition trade-offs.
The COLMAP documentation describes these differences; global mapping is **not** universally
faster or better for every dataset. Melkor does not repeat an unqualified "N× faster" figure:
any performance comparison between the incremental and global mappers must come from a benchmark
run on named datasets and hardware, recorded under `benchmarks/`, not from folklore. Choose the
mapper based on your data and, where it matters, on a measurement you can reproduce.

## Status in Melkor

The two COLMAP adapters — `colmap-incremental` and `colmap-global` — are delivered by the pinned
adapter protocol and the reproducible stage runner (work package 18). Each pins an exact COLMAP
identity, verifies that the build exposes the command it needs, validates the resulting model
with a parser rather than trusting the exit code, and records the run in an immutable manifest.
Until that adapter is qualified end-to-end, treat this document as the manual procedure; the
standalone GLOMAP installer is gone regardless.
