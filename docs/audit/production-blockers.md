# v2.0.0 production blockers

This is the authoritative blocker register for the `v2.0.0` release. It is copied from the
production-hardening review of 2026-07-14 and is the list the release gate checks against.

Severity meanings:

- **P0** — can cause incorrect output, data loss, security exposure, impossible installation, or
  a materially false production claim. No release candidate may be cut while one is open.
- **P1** — major ecosystem, portability, quality, or maintainability defect. No final release may
  be cut while one is open.
- **P2** — important hardening or governance work. May be deferred only with a written, public
  limitation and a named owner and date.

Status values: `open`, `in-progress`, `closed`. A blocker moves to `closed` only when its
acceptance evidence is attached — a passing test, a published artifact, or a settings change with
proof. "It should work" is not evidence.

| P0-10 | glTF ecosystem | Required glTF extensions are rejected broadly, and `KHR_gaussian_splatting` is not implemented. | Implement the pinned release-candidate specification, required-extension semantics, and conformance fixtures. | WP09 | in-progress — the exact Khronos `KHR_gaussian_splatting` RC spec (commit `63770cc`, 2026-04-15) is pinned in `third_party/specifications.lock.json` with the README and schema vendored and hash-verified in CI; the semantic decisions (SH degree 0–3, attribute names, linear scale/opacity, colour spaces, POINTS mode) are captured in `profiles/gltf/khr-gaussian-splatting-rc-63770cc.json`; the update procedure is documented. The **layout core** (`include/melkor/format/gltf_khr.hpp`, `src/formats/gltf_khr.cpp`) is now implemented and unit-tested (165 checks, `test_gltf_khr`): SH attribute naming and the m-ordering (COEF_0..2l = m=−l..+l), the flat↔address pyramid mapping, the spec's `C = R·diag(s)` matrix cross-checked element-wise against the vendored formula and shown to satisfy `C·Cᵀ = R diag(s²) Rᵀ`, and the two defined colour spaces with unknown-string rejection. The **GLB container framing** (`include/melkor/format/glb_container.hpp`, `src/formats/glb_container.cpp`) is implemented and adversarially tested (32 checks, `test_glb_container`, clean under ASan+UBSan): a strict, overflow-safe parser (checked arithmetic on every offset; a lying total length, an overflowing/misaligned chunk length, a straddling chunk header, JSON-not-first, duplicate/missing JSON all produce a clean failure with no over-read) and a `build_glb` that applies the mandated 4-byte padding and round-trips with the parser. The **accessor decoder** (`include/melkor/format/gltf_accessor.hpp`, `src/formats/gltf_accessor.cpp`) is implemented and tested (40 checks, `test_gltf_accessor`, clean under ASan+UBSan): it decodes float and signed/unsigned 8/16/32-bit integer components (with the exact glTF normalized rules — unsigned `c/max`, signed `max(c/max, −1)`), honours interleaved byte strides and byte offsets, and bounds-checks count/stride/offset with checked arithmetic so an overflowing or out-of-range accessor fails cleanly instead of over-reading. The **JSON document parser** (`include/melkor/format/gltf_document.hpp`, `src/formats/gltf_document.cpp`) is implemented and tested (46 checks, `test_gltf_document`, clean under ASan+UBSan): exception-free parsing of the reduced glTF document the splat reader needs (accessors, bufferViews, buffers, meshes/primitives with the `KHR_gaussian_splatting` extension, nodes with matrix-or-TRS transforms, the default scene, extension declarations), with every reader-followed index bounds-checked at parse so a successfully-parsed document has a self-consistent reference graph, and clean rejection of malformed JSON, wrong-typed fields, negative/out-of-range indices, an unsupported accessor type, and an incomplete extension. The **node transforms** (`include/melkor/format/gltf_transform.hpp`, `src/formats/gltf_transform.cpp`) are implemented and tested (14 checks, `test_gltf_transform`): a node's local transform from either the column-major `matrix` or the T·R·S triple (reusing the math oracle's quaternion handling so it agrees with the covariance transform), and the parent-chain affine composition — cross-checked by applying to points, and pinning that the column-major matrix is not read transposed and that the T·R·S order is correct. This produces exactly the `M` and `t` the reader feeds into `Σ' = MΣMᵀ` and `μ' = Mμ + t`. **Remaining:** the reader assembly (walk the scene graph, resolve+decode the splat attributes into a `SplatData`, apply the transform and SH rotation, emit a loss report), The **extension policy** (`include/melkor/format/gltf_extensions.hpp`, `src/formats/gltf_extensions.cpp`) is implemented and tested (17 checks, `test_gltf_extensions`): it rejects only the *required* extensions Melkor cannot honour (the P0-10 fix — the old behaviour rejected required extensions too broadly) and reports used-but-not-required unsupported ones as safely ignored, de-duplicated and deterministically ordered. The **accessor resolver** (`include/melkor/format/gltf_resolve.hpp`, `src/formats/gltf_resolve.cpp`) is implemented and tested (14 checks, `test_gltf_resolve`, clean under ASan+UBSan): it composes the bufferView+accessor offsets and validates the accessor's byte extent lies within its bufferView within its buffer — byte-extent checks the index-level parser does not do — then decodes, handling interleaved strides and rejecting an unavailable (external) buffer. The **per-primitive reader** (`include/melkor/format/gltf_reader.hpp`, `src/formats/gltf_reader.cpp`) is implemented and tested (27 checks, `test_gltf_reader`, clean under ASan+UBSan): it reads one KHR primitive into a local-space validated `SplatData`, checking POINTS mode and the `ellipse` kernel, the required attributes and their element types and consistent count, SH degree completeness/contiguity, and — the load-bearing part — the transpose from KHR's per-coefficient accessors (coefficient-major across splats) into the scene model's splat-major blocks, pinned by a degree-1 case with distinct per-coefficient values. An unknown colour space is surfaced as assumed, not silently guessed. The **scene-graph walk** (`read_gaussian_scene`) is implemented and tested (21 checks, `test_gltf_scene`, clean under ASan+UBSan): it gates unsupported required extensions as a hard error, walks the default scene cycle-safely (visited-set) composing node transforms, applies each instantiating node's transform to the geometry (mean via `μ' = Mμ + t`, covariance via `Σ' = MΣMᵀ` through the math oracle), merges primitives of differing SH degree by zero-padding to the maximum, and emits a loss report. The deferred Wigner-D SH rotation is handled honestly: a rotating node over degree≥1 SH raises a **severe, approvable** `LOSS_SH_ROTATION_NOT_APPLIED` (degree-0 or pure-scale/translation nodes raise nothing), so view-dependent colour is never silently left wrong. The top-level `read_glb` ties the container framing to the scene reader (validate framing → parse JSON chunk → feed the BIN chunk as buffer 0 → merged `SplatData`), verified end-to-end with a real GLB built by `build_glb` (round-trip of transformed positions) and with truncated/garbage inputs failing cleanly (`test_gltf_scene`, 27 checks, clean under ASan+UBSan). The **writer** (`include/melkor/format/gltf_writer.hpp`, `src/formats/gltf_writer.cpp`) is implemented and tested by full round-trip (47 checks, `test_gltf_writer`, clean under ASan+UBSan): it serialises a `SplatData` into a KHR GLB (the reverse SH transpose, per-attribute float accessors with POSITION min/max, one identity node), and a write→read cycle returns every field — positions, scales, rotations, opacities, and the spherical harmonics — to float precision. A degree-4 source is written at the degree-3 profile ceiling with a severe, approvable `LOSS_SH_DEGREE_TRUNCATED`. Both directions of the codec are now implemented and verified. **Remaining:** a conformance corpus validated by the Khronos glTF Validator, and wiring the reader/writer into the format registry and CLI (WP09). |

## Rules

- No P0 or P1 item may be waived.
- A P2 deferral requires an owner, a target version, a public limitation statement, and approval.
- "Not applicable" requires a written reason, and two-person review when it affects a public
  claim.
- A stale result from a different commit does not qualify the release. Exact-SHA evidence items
  must name the final release commit.
